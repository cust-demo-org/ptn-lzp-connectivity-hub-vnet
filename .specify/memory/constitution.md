# PLZ Connectivity Hub VNet Pattern Constitution

## Purpose

This constitution governs the design, implementation, review, and
evolution of the **Platform Landing Zone (PLZ) Connectivity Hub
VNet** Terraform pattern. The pattern provisions hub virtual network
infrastructure aligned to the Azure Landing Zones (ALZ) and Microsoft
Cloud Adoption Framework (CAF) — including hub VNets, firewalls,
firewall policies, gateways, bastion, private DNS, and peering to
existing spoke VNets — that platform teams operate and spoke
workloads consume.

The pattern is built on top of the official AVM pattern module
[`Azure/avm-ptn-alz-connectivity-hub-and-spoke-vnet/azurerm`](https://registry.terraform.io/modules/Azure/avm-ptn-alz-connectivity-hub-and-spoke-vnet/azurerm/latest)
as its primary engine, supplemented by individual AVM resource modules
only when the core pattern does not cover a required capability.

Every contributor, reviewer, and consumer of this pattern MUST
treat this document as the authoritative source of truth. Where
ambiguity exists between this constitution and any other project
artifact, this constitution prevails.

## Scope & Non-Scope

### In Scope

- Hub virtual network(s) with configurable address spaces, DNS
  servers, DDoS protection, and subnets.
- Azure Firewall (Basic / Standard / Premium) with configurable
  IP configurations, management IP, and availability zones.
- Azure Firewall Policy with DNS proxy, intrusion detection, TLS
  inspection, threat intelligence, and explicit proxy support.
- Azure Bastion (Basic / Standard) with public IP and configurable
  features (copy-paste, tunneling, IP connect, etc.).
- VPN Gateway (site-to-site, point-to-site, local network gateways,
  active-active, BGP).
- ExpressRoute Gateway with circuit connections and peering.
- Private DNS Zones (private link zones, auto-registration,
  virtual network links, resolution policies).
- Private DNS Resolver (inbound/outbound endpoints, forwarding
  rulesets).
- DDoS Protection Plan (shared across hub networks).
- Network Security Groups (created as supplementary resources and
  associated with hub subnets by ID).
- NAT Gateways (created as supplementary resources and associated
  with hub subnets by ID).
- Route tables (auto-generated by the core pattern for firewall and
  user subnets, plus gateway route tables).
- VNet peering from hub to existing spoke virtual networks.
- Log Analytics Workspace (provisioned by this pattern or consumed
  as a BYO input for diagnostics and firewall insights).
- Diagnostic settings for supplementary resources (NSGs, NAT
  Gateways).
- Network Watcher and flow-log configuration.
- Tags inherited globally and overridable per resource.

### Non-Scope

- Spoke virtual network creation (spokes are provisioned by their
  own landing zone patterns; this module peers TO existing spokes).
- Application-level workload infrastructure.
- Key Vaults, Storage Accounts, Managed Identities, Backup Vaults,
  Recovery Services Vaults (these belong in spoke/shared-services
  patterns).
- Firewall Rule Collection Groups, Application Rules, Network Rules,
  NAT Rules (managed separately outside this pattern to allow
  independent lifecycle).
- Entra ID tenant configuration.
- Management group hierarchy or Azure Policy definitions.
- Billing, cost management, or subscription vending.
- CI/CD pipeline definition (pipelines invoke this pattern but
  are not governed by it).

## Architecture: Core vs Supplementary Modules

### Core Pattern Module (Primary Engine)

| Module | Registry Source | Purpose |
|--------|----------------|---------|
| Hub & Spoke VNet | `Azure/avm-ptn-alz-connectivity-hub-and-spoke-vnet/azurerm` | Hub VNets, subnets, firewall, firewall policy, bastion, VPN GW, ER GW, private DNS zones, private DNS resolver, DDoS protection, route tables |

The core pattern module is the **single source of truth** for all
resources it supports. This wrapper pattern MUST NOT duplicate or
re-implement any capability the core pattern already provides.

The core pattern manages the following internal AVM modules:
- `Azure/avm-res-network-bastionhost/azurerm`
- `Azure/avm-res-network-publicipaddress/azurerm`
- `Azure/avm-res-network-ddosprotectionplan/azurerm`
- `Azure/avm-res-network-dnsresolver/azurerm`
- `Azure/avm-res-network-routetable/azurerm`
- `Azure/avm-res-network-privatednszone/azurerm`
- `Azure/avm-ptn-network-private-link-private-dns-zones/azurerm`
- Internal submodules for hub-virtual-network-mesh and virtual-network-gateway

**Do NOT add standalone AVM resource modules for any of the above.**

### Supplementary AVM Resource Modules

These modules fill gaps where the core pattern does not support
the resource or does not create the resource (only accepting IDs):

| Module | Registry Source | Justification |
|--------|----------------|---------------|
| Network Security Group | `Azure/avm-res-network-networksecuritygroup/azurerm` | Core pattern accepts NSG **IDs** on subnets but does not create NSGs |
| NAT Gateway | `Azure/avm-res-network-natgateway/azurerm` | Core pattern accepts NAT Gateway **IDs** on subnets but does not create NAT Gateways |
| Log Analytics Workspace | `Azure/avm-res-operationalinsights-workspace/azurerm` | Core pattern accepts LAW IDs for firewall insights but does not create a workspace |
| Network Watcher (Flow Logs) | `Azure/avm-res-network-networkwatcher/azurerm` | Core pattern does not manage network watcher or flow logs |
| VNet Peering (hub→spoke) | `Azure/avm-res-network-virtualnetwork/azurerm` peering submodule OR `azurerm_virtual_network_peering` | Core pattern manages mesh peering between hubs but does not peer to external spoke VNets |

Before adding ANY new supplementary module, the contributor MUST:
1. Verify the capability is not already provided by the core pattern.
2. Check the core pattern's latest release notes for newly added features.
3. Document the justification in the PR and in this table.

### Key-Based Cross-Resource Referencing

Following the learnings from `ptn-lza-ntwk-shared-services`, this
pattern MUST use map-key-based references (not raw resource IDs)
for wiring resources together within the module:

- NSGs MUST be referenced by `nsg_key` in subnet definitions.
- NAT Gateways MUST be referenced by `nat_gateway_key` in subnet definitions.
- Spoke VNets for peering MUST be referenced by `spoke_key` in the
  peering definitions (with the spoke's `resource_id` as the value).
- The Log Analytics Workspace MUST be referenceable by key or
  supplied as a BYO resource ID.
- Flow log storage accounts (if needed) MUST be referenceable by
  key or supplied as a BYO resource ID.

The pattern's `main.tf` resolves these keys to resource IDs via
`locals` blocks before passing them to module inputs.

## Core Principles

### I. Terraform-Only Declarative Infrastructure (NON-NEGOTIABLE)

All infrastructure MUST be expressed exclusively in HashiCorp
Configuration Language (HCL) executed by Terraform.

- Contributors MUST NOT introduce Bash, PowerShell, Python, or
  any other scripting language into the module source tree.
- The use of `null_resource`, `local-exec`, `remote-exec`, and
  any provider-level provisioner blocks is FORBIDDEN.
- External data retrieval MUST use only Terraform data sources
  or provider-native resources — never external scripts.
- Terraform version constraints MUST be pinned to `~> 1.12`
  (matching the core pattern's requirement).
- Provider constraints: `azurerm ~> 4.0`, `azapi ~> 2.4`
  (matching the core pattern), plus `random ~> 3.5` if needed.

**Rationale:** Declarative-only infrastructure ensures
reproducibility, auditability, and eliminates an entire class
of imperative side-effect bugs.

### II. Azure Verified Modules Exclusive — Core Pattern First (NON-NEGOTIABLE)

The core AVM pattern module `avm-ptn-alz-connectivity-hub-and-spoke-vnet`
MUST be the primary engine for all resources it supports.

- Before adding a supplementary AVM resource module, the
  contributor MUST verify that the core pattern does not already
  cover the capability. This verification MUST be documented in
  the PR.
- Supplementary AVM resource modules are permitted ONLY when the
  core pattern does not create or manage the resource type.
- Custom `azurerm_*` resource blocks are permitted ONLY when no
  AVM module (pattern or resource) exists for the resource type
  AND the omission is documented with a tracking issue for future
  AVM adoption.
- AVM module versions MUST be pinned to an exact version and
  updated deliberately through a reviewed pull request.
- When the core pattern adds support for a resource this wrapper
  currently handles via a supplementary module, the supplementary
  module MUST be removed in favour of the core pattern's
  implementation.

**Rationale:** The core pattern is maintained by the ALZ team
with integrated testing. Using it maximises supportability and
reduces the surface area of custom code. Supplementary modules
address gaps only.

### III. Configuration-Driven Reusability

The pattern MUST be fully parameterised so that consumers can
deploy it across environments, subscriptions, regions, and
teams without modifying module source code.

- The core pattern module's `hub_virtual_networks` variable is a
  map, inherently supporting multi-hub topologies. This wrapper
  MUST preserve that capability.
- Supplementary resources (NSGs, NAT Gateways) MUST also be
  map-based variables to support multiple instances.
- Variables MUST default to AVM/core-pattern defaults wherever
  possible. Opinionated defaults MUST be documented in the
  variable's `description` and MUST be secure-by-default.
- Variables MUST NOT use overly broad types (e.g., `any`).
  Prefer `object({...})`, `map(...)`, `list(...)`, or scalar
  types with explicit type constraints.
- Sensitive values MUST be marked with `sensitive = true`.
- The pattern MUST support toggling optional components via the
  core pattern's `enabled_resources` flags (firewall, bastion,
  VPN GW, ER GW, private DNS, DNS resolver) — do NOT reimplement
  feature flags that the core pattern already provides.

**Rationale:** A configuration-driven approach enables a
single codebase to serve every environment and team, reducing
drift and duplication.

### IV. Security by Default (NON-NEGOTIABLE)

The pattern MUST enforce a secure-by-default network posture.
Security requirements are non-negotiable and override
convenience in every trade-off decision.

- **Least privilege:** RBAC assignments and NSG rules MUST follow
  the principle of least privilege. Broad wildcards (`*`) in
  network rules are FORBIDDEN unless explicitly justified.
- **Network segmentation:** NSGs MUST be associable with every
  user subnet. Default-deny is the baseline posture (Azure's
  built-in default rules satisfy this; the pattern does not
  auto-inject additional deny rules).
- **No public exposure by default:** The firewall, bastion, and
  gateway public IPs are inherently public (required for
  functionality). All other resources MUST NOT expose public
  endpoints by default.
- **Encryption:** All data-in-transit MUST use TLS 1.2+.
  Firewall policy TLS inspection settings MUST be exposed.
- **Logging and auditing:** Diagnostic settings MUST be enabled
  for every supplementary resource that supports them. Logs
  MUST be sent to the Log Analytics workspace. The core pattern
  handles diagnostics for its own managed resources internally.
- **DDoS Protection:** The pattern MUST support DDoS Protection
  Plan association via the core pattern's built-in support.
- **Default outbound access:** All subnets SHOULD default to
  `default_outbound_access_enabled = false` (matching the core
  pattern's defaults).

**Rationale:** A hub network is the trust boundary for all
workloads. A single misconfiguration has blast-radius across
every spoke.

### V. Reliability & Determinism (NON-NEGOTIABLE)

The pattern MUST produce deterministic, repeatable deployments
and support safe, incremental updates.

- **Deterministic plans:** Given identical inputs and state, a
  `terraform plan` MUST produce an identical execution plan.
- **Safe updates:** Breaking changes to variable interfaces
  MUST follow semantic versioning (see Governance).
- **Resiliency:** Zone-redundancy options MUST be exposed for
  firewall, bastion, and gateway resources via the core
  pattern's variables. NAT Gateways MUST support zone
  configuration.
- **State safety:** The pattern MUST NOT use `terraform taint`,
  `terraform state rm`, or any state-manipulation commands as
  part of normal operation.
- **Idempotency:** Consecutive `terraform apply` runs with
  unchanged inputs MUST result in zero resource changes.
- **Retry support:** The core pattern's `retry` and `timeouts`
  variables MUST be exposed to handle transient provisioning
  errors.

**Rationale:** Hub infrastructure underpins all workloads.
Non-deterministic or unsafe deployments risk cascading
outages.

### VI. Tooling-Based Documentation (NON-NEGOTIABLE)

All Terraform module documentation (inputs, outputs, providers,
resources, sub-modules) MUST be generated by
[terraform-docs](https://terraform-docs.io/).

- AI-generated or manually curated Terraform input/output
  documentation is FORBIDDEN.
- Each module directory MUST contain a `README.md` with
  `terraform-docs` injection markers.
- `terraform-docs` MUST be executed after any variable or output
  change.
- Every input variable MUST have a non-empty `description`
  attribute following AVM-style conventions:
  - Opening sentence states the purpose.
  - Nested attributes are documented inline.
  - Key-based referencing is explained with examples.
  - Security implications are called out explicitly.

**Rationale:** Tooling-generated documentation is always in
sync with the code. Manual documentation drifts.

### VII. Operability & Observability

The pattern MUST be operable across environments,
subscriptions, and teams without bespoke knowledge.

- **Meaningful outputs:** The root module MUST export:
  - All outputs from the core pattern (hub VNet IDs/names,
    firewall IPs, bastion IDs, DNS zone IDs, gateway IDs, etc.).
  - Supplementary resource outputs (NSG IDs, NAT Gateway IDs,
    LAW ID, flow log IDs).
  - Peering resource IDs for hub→spoke connections.
- **Output descriptions:** Every output MUST have a non-empty
  `description` attribute.
- **Naming:** The core pattern provides built-in naming
  conventions with `${location}` and `${sequence}` placeholders.
  This wrapper MUST expose the `default_naming_convention`
  variable to allow overrides.
- **Tagging:** All resources MUST support a common `tags` variable,
  propagated to the core pattern and supplementary modules.

**Rationale:** Operations teams must be able to onboard,
troubleshoot, and extend the pattern without reverse-
engineering module internals.

### VIII. Extensibility & Composability

The pattern MUST be designed for extension without modification.

- New optional supplementary modules MUST be addable by providing
  new input variables and module blocks — never by editing existing
  resource blocks.
- The pattern MUST expose outputs that downstream patterns (e.g.,
  spoke shared-services modules) need for peering, DNS linking,
  and routing.
- Cross-pattern integration points MUST be exposed as input/output
  variable contracts, not hard-coded resource references.
- The pattern MUST NOT hard-code subscription IDs, tenant IDs,
  resource group names, or region. These MUST be supplied via
  variables or inherited from the provider configuration.
- When the core pattern releases new capabilities that overlap
  with supplementary modules, migration MUST be planned using
  `moved` blocks to avoid state churn.

**Rationale:** Landing zone patterns evolve. Extensibility
ensures the pattern can grow without forking.

## Quality Gates & Definition of Done

### Mandatory Quality Gates

1. **`terraform fmt -check -recursive`** — All HCL files MUST
   be formatted.
2. **`terraform validate`** — The configuration MUST pass
   validation with no errors and no warnings (root module plus
   all examples).
3. **`terraform plan`** — A plan MUST be generated successfully
   against a representative environment.
4. **`terraform-docs` freshness** — Generated documentation
   MUST be up to date.
5. **Linting (tflint or equivalent)** — A Terraform linter
   MUST be executed.

### Prohibited Gate Implementations

- Quality gates MUST NOT be implemented as Bash, PowerShell,
  or Python wrapper scripts.
- `null_resource` or `local-exec` MUST NOT be used for any
  validation or gate logic.

### Definition of Done / Acceptance Criteria

A change is considered **conformant** when ALL of the following
are met:

1. All five mandatory quality gates pass with zero errors.
2. Every Azure resource is provisioned via the core pattern or
   a documented supplementary AVM module. No undocumented
   custom `azurerm_*` resources exist.
3. No `null_resource`, `local-exec`, `remote-exec`, provisioner,
   or external script exists in the codebase.
4. All input variables have type constraints, descriptions, and
   secure defaults. No variable uses type `any`.
5. All outputs have descriptions and expose the IDs/attributes
   required by documented consumer contracts.
6. Diagnostic settings are enabled for every supplementary
   resource that supports them.
7. No resource exposes a public endpoint unless functionally
   required (firewall, bastion, gateway public IPs).
8. Zone-redundancy options are configured where supported.
9. `terraform plan` with unchanged inputs produces zero changes.
10. `terraform-docs`-generated README sections are current.
11. Supplementary modules are justified (core pattern does not
    support the capability).
12. Key-based cross-resource referencing is used (no raw IDs in
    variable interfaces where key-based wiring is possible).

## Governance

This constitution supersedes all other project guidance,
conventions, or informal practices. In any conflict, the
constitution prevails.

- **Amendment process:** Any change to this constitution MUST
  be proposed via a pull request, reviewed by at least one
  maintainer, and approved before merge.
- **Versioning:** Semantic Versioning:
  - **MAJOR** — Backward-incompatible principle removal or
    redefinition.
  - **MINOR** — New principle or materially expanded guidance.
  - **PATCH** — Clarifications, wording, or typo fixes.
- **Core pattern tracking:** When the core AVM pattern module
  releases a new version, this pattern MUST evaluate the release
  notes for:
  - New capabilities that obsolete supplementary modules.
  - Breaking changes that require variable interface updates.
  - New `enabled_resources` flags to expose.
  The update assessment MUST be documented in the PR.
- **Compliance review:** Every PR MUST include a self-assessment
  checklist confirming compliance with each NON-NEGOTIABLE
  principle.
- **Exception process:** Exceptions to NON-NEGOTIABLE principles
  MUST be documented in an ADR in `docs/decisions/`, approved by
  at least two maintainers, and tracked as technical debt.
- **Periodic review:** The constitution SHOULD be reviewed at
  least once per quarter — especially when the core pattern
  releases a major version.
